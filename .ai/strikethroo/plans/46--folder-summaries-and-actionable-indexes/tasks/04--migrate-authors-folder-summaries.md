# Extend the v1→v2 migrate clustering to author a folder `summary` per created folder

## Objective
At the existing quarantined LLM clustering moment of the v1→v2 migration, emit a
one-line `summary` per topical folder the clustering creates, and have the
deterministic placement primitive write that summary into the new folder's
`index.md` frontmatter — so any v1 repo (and this repo via revert-and-re-migrate)
gets authored folder summaries through the real path, while the deterministic
core stays summary-free.

## Skills Required
- `typescript`: extend the `ClusterFn`/`Placement` contract and `CLUSTER_INSTRUCTIONS` in `src/commands/migrate.ts`, and the write primitive in `src/lib/migrate-flat-to-tree.ts`.
- `vitest`: extend `tests/commands/migrate.test.ts` using the deterministic cluster override (no harness spawned).

## Acceptance Criteria
- [ ] The clustering result carries a `summary` per created folder (a `folder -> summary` mapping alongside the existing `id -> targetFolder` placements). The harness `CLUSTER_INSTRUCTIONS` and `PlacementResponseSchema` are extended to request and parse it.
- [ ] The deterministic write path (`writePlacements`, or a sibling step invoked by `flatToTreeStep`) stamps each folder's authored `summary` into that folder's `index.md` frontmatter using `IndexFrontmatterSchema` (the field added in Task 1).
- [ ] A folder summary is authored to read as a sentence fragment that completes `for more information on …` / `to learn about: …` (phrasing contract enforced in the prompt text).
- [ ] Running the extended migrate on a flat fixture with an injected deterministic cluster writes a non-empty `summary` into every created folder's `index.md` frontmatter (Success Criterion 4).
- [ ] The test path uses the existing `cluster` override injection (`MigrateOptions.cluster`) so the suite never spawns a real harness (`requiresHarness` stays false under the override).
- [ ] Identity/placement invariants are unchanged: ids preserved, all-or-nothing write, no git.

Use your internal Todo tool to track these and keep on track.

## Technical Requirements
- `flatToTreeStep` (`src/commands/migrate.ts:92`) calls `cluster(leaves)` → `reconcilePlacements` → `writePlacements`. The `summary` must flow from the cluster result through to the index write.
- `ClusterFn` is `(leaves: FlatLeaf[]) => Placement[] | Promise<Placement[]>` (`:20`). It must additionally surface per-folder summaries. Options: return `{ placements, folderSummaries: Record<folder,string> }`, or extend `Placement`/add a parallel structure. Choose the smallest coherent change and update `reconcilePlacements`, `makeHarnessCluster`, `parsePlacements`, and `PlacementResponseSchema` accordingly.
- After placements are written, the new folders exist but their `index.md` is generated by the subsequent `index rebuild`/`generateIndex`. The authored summary must be written into the folder `index.md` frontmatter so the very next `generateIndex` self-preserves it (Task 1). Decide whether to (a) write a minimal `index.md` stamping just the summary right after placement, or (b) thread the `folderSummaries` into the rebuild that the migrate flow triggers. Prefer (a): write the summary into the folder's `index.md` frontmatter immediately so self-preserve carries it on the next rebuild.

## Input Dependencies
- Task 1: `IndexFrontmatterSchema.summary` and the self-preserve carry must exist so the authored summary persists past the first rebuild.

## Output Artifacts
- An extended migrate authoring path that satisfies Success Criterion 4 and is the mechanism the dogfood backfill (Task 7) relies on.

## Implementation Notes
This is one of exactly two sanctioned authoring moments; the determinism
boundary does not move. The LLM invents the summary; deterministic code only
writes/carries it.

<details>
<summary>Detailed implementation guidance</summary>

1. **Extend the cluster contract.** Decide the shape. Recommended:
   ```ts
   export interface ClusterResult {
     placements: Placement[];            // id -> targetFolder (as today)
     folderSummaries: Record<string, string>; // folder -> one-line summary
   }
   export type ClusterFn = (leaves: FlatLeaf[]) => ClusterResult | Promise<ClusterResult>;
   ```
   Update `flatToTreeStep` to consume `folderSummaries`, `reconcilePlacements`
   (unchanged for placements; ignore summaries), and the test stubs in
   `tests/commands/migrate.test.ts` to return the new shape.

2. **Harness prompt + schema.** In `CLUSTER_INSTRUCTIONS` (`migrate.ts:139`), add:
   request a `summary` per folder, phrased to complete `for more information on
   <summary>` — i.e. a noun phrase / sentence fragment, lowercase start, no
   trailing period, ≤ ~140 chars. Change the response shape to
   `{"placements":[...],"folders":[{"folder":"<folder>","summary":"<text>"}]}` and
   extend `PlacementResponseSchema` + `parsePlacements` to parse it.

3. **Write the summary.** After `writePlacements` creates the folders, write each
   `folderSummaries[folder]` into `nodes/<folder>/index.md` frontmatter. Reuse
   `IndexFrontmatterSchema` and `gray-matter`. The simplest robust approach:
   immediately after placement, for each folder with a summary, write a minimal
   `index.md` whose frontmatter carries `schema_version`, a placeholder
   `nodes_hash`/`node_count`, and the `summary`; the next deterministic rebuild
   regenerates the body and self-preserves the `summary`. Confirm the rebuild that
   the migrate flow (or the caller) runs reads this file before overwriting
   (Task 1 harvest), so the value survives.
   - If writing a placeholder `index.md` risks a transient invalid `nodes_hash`,
     instead stamp the summary into the folder index produced by the rebuild by
     threading `folderSummaries` into that one rebuild call. Pick whichever keeps
     `generateIndex` the single self-preserve owner; document the choice.

4. **Tests** (`tests/commands/migrate.test.ts`):
   - Inject a deterministic `cluster` override returning fixed placements AND
     fixed `folderSummaries`. Run the migrate step against a flat v1 fixture.
   - Assert each created folder's `index.md` frontmatter has a non-empty
     `summary` equal to the injected value (Success Criterion 4).
   - Assert `requiresHarness` is false when the override is present (no harness
     spawn).
   - Assert ids and edges are preserved (existing invariants still hold).

**Test philosophy — "write a few tests, mostly integration".** Meaningful tests
verify custom business logic, critical paths, and edge cases specific to this
application. Test *your* code, not the framework. WRITE tests for: custom
business logic and algorithms; critical workflows and data transformations; edge
cases and error conditions for core functionality; integration points; complex
validation. Do NOT write tests for: third-party library functionality; framework
features; simple CRUD without custom logic; trivial getters/setters or static
config; obvious functionality that would break immediately if incorrect. The
authoring-to-frontmatter flow via the deterministic override is the integration
point worth covering; do not test the harness exec path (it is never run in CI).
