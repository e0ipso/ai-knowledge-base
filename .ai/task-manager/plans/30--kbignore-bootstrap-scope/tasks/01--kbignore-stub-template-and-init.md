# Generate `.kbignore` stub from harness registry and emit it from `init`

## Objective

Produce a default `.kbignore` for repo root that is simultaneously the user-facing scope configuration and the migration target for the implicit `harnessInstructionSkipPatterns`. Wire `init` (and `init --upgrade`) to write it once when absent, never overwrite.

## Skills Required

- typescript

## Acceptance Criteria

- [ ] An emit routine produces a `.kbignore` string containing: a header comment block (purpose, gitignore-style syntax, repo-root relative paths, what is auto-skipped regardless: `.git/`, `node_modules/`, `STATIC_SKIPS`); a worked example showing directory deny + glob deny + `!`-prefixed un-ignore (in the order that respects the parent-directory caveat) and a one-line warning about that caveat; a commented-out "common noise" block (e.g. `build/`, `dist/`, `coverage/`, `**/*.generated.md`); an uncommented deny block listing every harness instruction directory currently produced by `harnessInstructionSkipPatterns` (`.claude/skills/`, `.claude/commands/`, `.claude/hooks/`, `.claude/plugins/`, and the per-adapter equivalents for codex, cursor, opencode).
- [ ] The deny block is generated by enumerating registered harness adapters (so adding a new adapter expands the stub automatically next time `init` runs).
- [ ] `init` writes the stub to `<repoRoot>/.kbignore` when absent; skips when present.
- [ ] `init --upgrade` writes the stub when absent; skips when present. Never overwrites.
- [ ] Stub emission runs immediately after `updateGitignore` inside `runInit` so the review surface is contiguous.
- [ ] Unit tests cover: file written when absent on fresh init; file untouched when already present (custom content preserved); file written on `--upgrade` when absent; deny block enumerates every registered harness adapter's instruction directories.
- [ ] `npm run build` and `npm test` pass.

Use your internal Todo tool to track these and keep on track.

## Technical Requirements

- New module (e.g. `src/lib/kbignore-stub.ts`) exporting `renderKbignoreStub(adapters: HarnessAdapter[]): string` and `ensureKbignore(repoRoot: string): { written: boolean; path: string }`.
- Edits to `src/commands/init.ts` (`runInit` flow around line 80) to invoke `ensureKbignore(repoRoot)`.
- Tests in `tests/commands/init.test.ts` (extend if exists, otherwise create).

## Input Dependencies

None.

## Output Artifacts

- `renderKbignoreStub` / `ensureKbignore` exports used by Task 4 (doctor) for symmetry and by Task 5 (docs) as the canonical stub reference.
- A `.kbignore` file emitted on `init` for downstream Task 2 (discovery) and Task 3 (CLI/command).

## Implementation Notes

<details>
<summary>Guidance</summary>

- Render the stub at runtime from the harness registry rather than checking in a static template file — the plan calls out that adding a future adapter should not require hand-editing the stub.
- Pull adapter instruction directory names by iterating `listHarnessIds()` → `getHarness(id).paths(repoRoot)` and converting `skillsDir`/`hooksDir`/etc. to repo-root-relative posix forms with a trailing slash (gitignore directory semantics).
- The worked example must demonstrate the correct ordering for `!` un-ignoring under an excluded ancestor — verify against the gitignore spec, not from memory. Example pattern: deny `docs/` then re-include `!docs/AGENTS.md` will NOT work; you must un-ignore the directory itself first. Make this explicit in the comment.
- `STATIC_SKIPS` is the always-on set from `src/lib/bootstrap.ts` — mention it by name in the header so users know editing `.kbignore` will not bypass things like `.git/` or `node_modules/`.
- Do NOT touch `harnessInstructionSkipPatterns` itself in this task — its removal is Task 2's responsibility. This task only adds the new surface.
- Match `init.ts`'s existing logging/return shape conventions (look at how `updateGitignore` reports its action) for consistency.
- Test the emit determinism: calling `renderKbignoreStub` twice with the same adapter list returns byte-identical output (sort adapter ids).

</details>
